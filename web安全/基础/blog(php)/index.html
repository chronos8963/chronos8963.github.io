0x3.漏洞的利用方式

graph LR;
    A(漏洞利用)-->B(失能)
    A(漏洞利用)-->C(读取)
    A(漏洞利用)-->D(写入)
    A(漏洞利用)-->E(执行)
    B-->F(服务宕机)
    C-->G(读敏感文件)
    C-->H(读数据库信息)
    D-->I(篡改数据)
    D-->J(提权)
    D-->K(加密硬盘)
    E-->L(执行命令)
    E-->M(内网渗透)
    E-->N(反弹shell)

失能

低级危害的利用方式，损人不利己，仅仅是让对方资产暂时失去服务功能，一般不会有其他的损失。更偏向对对方的名誉及形象进行破坏，技术上损失小，无形资产损失大。

读取

中级危害的利用方式，能够读取目标资产中的敏感数据，如数据库敏感信息，密码、不对外的文档文本等等。

写入

高级危害的利用方式，不仅能够读取目标中的敏感数据，还可以通过写入数据来篡改目标数据、修改目标的对外页面，例如，当年著名的伊朗网军攻陷百度事件，百度网页内容惨遭篡改。这种漏洞利用危害性、影响性巨大，属于高危漏洞。

执行

最高级别危害的利用方式，可以在目标资产上执行任意命令，完全控制目标资产，控制内部其他资产，不仅对数据洗劫一空，甚至可以操作目标指令，造成巨大损失。例如震网木马。

渗透中POC、EXP、Payload与Shellcode的区别

1. POC、EXP、Payload与Shellcode

POC：全称 ' Proof of Concept '，中文 ' 概念验证 ' ，常指一段漏洞证明的代码。

EXP：全称 ' Exploit '，中文 ' 利用 '，指利用系统漏洞进行攻击的动作。

Payload：中文 ' 有效载荷 '，指成功exploit之后，真正在目标系统执行的代码或指令。

Shellcode：简单翻译 ' shell代码 '，是Payload的一种，由于其建立正向/反向shell而得名。

2. 几点注意

POC是用来证明漏洞存在的，EXP是用来利用漏洞的，两者通常不是一类，或者说，POC通常是无害的，EXP通常是有害的，有了POC，才有EXP。

Payload有很多种，它可以是Shellcode，也可以直接是一段系统命令。同一个Payload可以用于多个漏洞，但每个漏洞都有其自己的EXP，也就是说不存在通用的EXP。

Shellcode也有很多种，包括正向的，反向的，甚至meterpreter。

Shellcode与Shellshcok不是一个，Shellshock特指14年发现的Shellshock漏洞。

3. Payload模块

在Metasploit Framework 6大模块中有一个Payload模块，在该模块下有Single、Stager、Stages这三种类型，Single是一个all-in-one的Payload，不依赖其他的文件，所以它的体积会比较大，Stager主要用于当目标计算机的内存有限时，可以先传输一个较小的Stager用于建立连接，Stages指利用Stager建立的连接下载后续的Payload。Stager和Stages都有多种类型，适用于不同场景。

4. 总结

想象自己是一个特工，你的目标是监控一个重要的人，有一天你怀疑目标家里的窗子可能没有关，于是你上前推了推，结果推开了，这是一个POC。之后你回去了，开始准备第二天的渗透计划，第二天你通过同样的漏洞渗透进了它家，仔细查看了所有的重要文件，离开时还安装了一个隐蔽的窃听器，这一天你所做的就是一个EXP，你在他家所做的就是不同的Payload，就把窃听器当作Shellcode吧！

0x1.文件包含的本质

文件包含，就是指定一个文件，将里面的内容包含进当前的脚本中执行。这样做的好处就是

  • 代码复用
  • 并行开发
  • 模块化
  • 可移植性

文件包含，是可以在代码中的任意位置包含的，并不非要是文件头包含

包含的内容，可以有php代码，包含后，会从磁盘读取这个文件所有内容，取到的内容插入到当前包含代码的位置

string.rot13

string.rot13（自 PHP 4.3.0 起）使用此过滤器等同于用 str_rot13()函数处理所有的流数据。

str_rot13—对字符串执行ROT13转换. ROT13编码简单地使用字母表中后面第13个字母替换当前字母，同时忽略非字母表中的字符。编码和解码都使用相同的函数，传递一个编码过的字符串作为参数，将得到原始字符串。

string.toupper

string.tolower（自 PHP 5.0.0 起）使用此过滤器等同于用 strtolower()函数处理所有的流数据。

strtoupper—将字符串转化为大写

string.tolower（自 PHP 5.0.0 起）使用此过滤器等同于用 strtolower()函数处理所有的流数据。

strtolower—将字符串转化为小写

string.strip_tags

使用此过滤器等同于用 strip_tags()函数处理所有的流数据。可以用两种格式接收参数：一种是和strip_tags()函数第二个参数相似的一个包含有标记列表的字符串，一种是一个包含有标记名的数组。

strip_tags—从字符串中去除 HTML 和 PHP 标记.该函数尝试返回给定的字符串str去除空字符、HTML 和 PHP 标记后的结果。它使用与函数fgetss()一样的机制去除标记。

convert.base64

convert.base64-encode和 convert.base64-decode使用这两个过滤器等同于分别用base64_encode()和base64_decode()函数处理所有的流数据。convert.base64-encode支持以一个关联数组给出的参数。如果给出了line-length，base64 输出将被用line-length个字符为 长度而截成块。如果给出了line-break-chars，每块将被用给出的字符隔开。这些参数的效果和用base64_encode()再加上 chunk_split()相同。

convert.quoted

现在唯一的问题就是我们不知道文件名字，而PHP的文件包含语法中，是不支持通配符的。必须完整的写出要包含的文件路径，才可以正确包含。

在给PHP发送POST数据包时，如果数据包里包含文件区块，无论你访问的代码中有没有处理文件上传的逻辑，PHP都会将这个文件保存成一个临时文件（通常是/tmp/php[6个随机字符]），文件名可以在$_FILES变量中找到。这个临时文件，在请求结束后就会被删除。同时，因为phpinfo界面会将当前请求上下文中所有变量都打印出来，所以我们如果向phpinfo界面发送包含文件区块的数据包，则即可在返回包里找到$_FILES变量的内容，自然也包含临时文件名。在文件包含漏洞找不到可利用的文件时，即可利用这个方法，找到临时文件名，然后包含之，但文件包含漏洞和phpinfo页面通常是2个页面。理论上，我们要先发送数据包给phpinfo界面，然后从返回界面中匹配出临时文件名，再将这个文件名发送给文件包含漏洞页面，进行getshell。在第一个请求结束时，临时文件就被删除了，第2个请求自然也就无法包含。这个时候就需要用到条件竞争，具体流程如下：

  • 发送包含了webshell的上传数据包给phpinfo界面，这个数据包的header、get等位置需要塞满垃圾数据

  • 因为phpinfo界面会将所有数据都打印下来，1中的垃圾数据会将整个phpinfo页面撑的非常大

  • php默认的输出缓冲区大小为4096，可以理解为php每次返回4096个字节给socket链接

  • 所以，我们直接操作原生socket，每次读取4096个字节。只要读取到的字符里包含临时文件名，就立即发送第二个数据包

  • 此时，第一个数据包的socket连接实际上还没结束，因为php还在继续每次输出4096个字节，所以临时文件此时还没有删除

  • 利用这个时间差，第二个数据包，也就是文件包含漏洞的利用，即可成功包含临时文件，最终getshell

0x1 php的文件上传机制

php使用临时文件移动的方式来上传文件，通过$_FILES数组包含文件信息，在PHP脚本处理过程中，对已经存在在临时目录的/tmp/php??????文件使用move_uploaded_file函数来移动到新的位置。

文件上传信息包含的数组$_FILES数组中，是支持多文件同时上传的。默认情况下，上传的名字为file,所以我们可以在$_FILES['file']这个数组中获得，我们可以获得以下信息

  • $_FILES['file']['name']获得上传文件的原始文件名
  • $_FILES['file']['tmp_name']上传的文件在临时目录中的文件名
  • $_FILES['file']['size']上传文件的大小，单位为字节
  • $_FILES['file']['type']上传文件的类型

在C程序中，字符串结束时，会有一个标记，表示字符串结束了，这个标记，设计人员规定改为\00字符，也就是0号字符。

由于PHP也是基于C语言开发出来的，那么在一定程度上，也会有00截断的问题。

所谓00截断，是指通过00字符，将字符串从非预期位置截断开来。比如我们上传的文件名字为123.php，这个明显是不让直接上传的，我们修改后缀为123.php%00.jpg，那么后台判断文件时，获取到的后缀就是符合规定的jpg后缀，然后在保存文件时文件路径为upload/123.php%00.jpg，由于00字符截断，最终保存的文件内容为upload/123.php后面的.jpg被截断舍弃造成了脚板上传代码执行。

由于漏洞非常古老，古老到我们几乎遇不到,只有PHP版本小于5.3.4；java版本小于7u40，才存在此漏洞。

1 配置文件绕过

.htaccess配置文件

前面我们使用了vscode来编辑php代码，在我们调整web根目录的时候，会自动生成两个文件，分别是

  • nginx.htaccess
  • .htaccess

内容都是空的，这里允许通过在脚本目录上传特定格式的配置文件，来修改PHP的解释器的各项功能

.htaccess和.user.ini的关系如下

.user.ini配置文件

apache的服务器二进制文件为httpd，那么它的配置文件一般为.htaccess

同样的，php.ini是可以控制php的各项解释执行参数的配置文件，他有一部分的参数值也是可以通过某个配置文件来修改的，这个文件名字可以在php.ini中找到

php作为解释器，是不直接对外接受文件的，需要一个web服务器来与他进行通信，告诉他要解析执行的文件路径和各项参数。目前php主流的解析方案有2种

  • Apache的模块执行
  • php-fpm的监听执行 FASTCGI模式

可以从官方文档中了解到，.user.ini只支持PHP_INI_PERDIR``PHP_INI_USER模式的设置

那么对于黑名单，不让上传php格式的文件的情况下，如果我们能上传.user.ini文件，也就可以通过auto_append_file参数来包含进当前的php文件中一起执行

使用.user.ini自动附加文件时，当前目录必须要有php文件

getimagesize绕过

在图片上传中，部分开发者认为，只要检测上传的内容是一张正确的图片（能够渲染）就认为不可能是恶意代码，于是开发后缀上传，只通过getimagesize函数来检测是否是正确的图片。

php的getimagesize函数默认是支持XBM格式的图片，但是XBM格式的图片，里面用字符串规定了高度和宽度的值，这里会逐行读取，如果出现类似#define %s %d的格式内容，会自动格式化字符串的值，最后如果width和height值不为0，就返回图片的高和宽。

基于这个特性，我们可以直接在上传的php文件中增加这两个变量的声明，来绕过getimagesize函数的检测。
